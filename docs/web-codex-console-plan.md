# Web Codex Console 方案草案

## 目标

做一个面向手机使用的 Web 管理端，用来管理这台电脑上的 Codex CLI 会话。

这个方案和当前 `telegram-codex-bot` 分开设计，TG 方案继续保留，Web 方案单独落地，避免两套交互逻辑互相污染。

## 设计原则

1. 不替代当前 TG 方案，先并行存在。
2. Web 端优先服务“手机查看和操作”，不是桌面 IDE 替代品。
3. 优先管理“会话”，不是直接暴露“任意 shell”。
4. 先做最小可用版本，再补实时输出、审批和文件管理。
5. 默认只给局域网、Tailscale 或本机访问，不直接裸露公网。

## 拆分方式

建议新建独立目录，例如：

```text
/
```

不要直接把 Web 服务塞进当前 TG 机器人里。原因：

1. TG 机器人是消息桥接模型。
2. Web 控制台是长连接和状态管理模型。
3. 两者的状态同步、权限边界、前端需求完全不同。

## 第一阶段目标

先做一个最小版本，只覆盖下面几件事：

1. 查看最近 Codex 会话列表。
2. 新建一个会话。
3. 进入某个会话继续发送消息。
4. 实时查看会话状态和最终回复。
5. 展示截图、图片、文档等产物。
6. 支持停止当前任务。

这一阶段先不做复杂账号体系，也不做多用户。

## 推荐架构

### 后端

技术建议：

1. `Node.js`
2. `Fastify` 或 `Express`
3. `SSE` 或 `WebSocket` 做实时更新

后端职责：

1. 管理 Codex 会话生命周期。
2. 调用 `codex exec --json` / `codex exec resume`。
3. 维护本地 session 状态。
4. 暴露会话、消息、审批、文件下载接口。
5. 统一做安全校验。

### 前端

技术建议：

1. `Vue 3 + Vite`
2. 移动端优先布局
3. 不追求复杂组件库，先把信息密度和操作路径做顺

前端页面建议：

1. 会话列表页
2. 会话详情页
3. 文件产物区
4. 设置页

## 后端核心模块

### 1. Session Manager

负责：

1. 创建本地 session
2. 绑定 `codexThreadId`
3. 保存最近回复、最近错误、执行状态
4. 恢复历史会话

建议持久化：

1. `runtime/web-sessions.json`
2. 必要时拆成每会话一个 json 文件

### 2. Codex Runner

负责：

1. 组装 `codex exec` 参数
2. 解析 `stdout` 的 `json event`
3. 把事件转成前端可消费的统一事件
4. 处理异常、超时、停止

### 3. Artifact Manager

负责：

1. 收集截图和文件产物
2. 识别图片、文档、压缩包
3. 提供下载接口
4. 提供缩略图或预览地址

### 4. Approval Manager

如果后续接 `codex app-server`，这里再扩展：

1. 待审批事件
2. 批准
3. 拒绝

第一阶段如果继续沿用 `exec/resume`，可以先留空。

## 前端交互草案

### 会话列表页

展示：

1. 标题
2. 最近问题摘要
3. 当前状态
4. 最近活动时间

操作：

1. 新建会话
2. 打开会话
3. 删除本地记录

### 会话详情页

展示：

1. 当前状态
2. 耗时
3. 实时事件流
4. 最终回复
5. 相关文件和图片

操作：

1. 发送消息
2. 停止执行
3. 重新发送
4. 切换到历史会话

### 文件展示

规则建议：

1. 图片直接预览
2. 文档显示文件名和下载按钮
3. 文本文件支持预览前几百行

## API 草案

### 会话

```text
GET    /api/sessions
POST   /api/sessions
GET    /api/sessions/:id
POST   /api/sessions/:id/messages
POST   /api/sessions/:id/stop
```

### 实时事件

```text
GET    /api/sessions/:id/stream
```

### 文件

```text
GET    /api/files/:id
```

## 实时事件模型

建议统一成前端友好的事件结构：

```json
{
  "type": "status|message|artifact|error|done",
  "sessionId": "xxx",
  "timestamp": "2026-03-18T08:00:00.000Z",
  "payload": {}
}
```

其中：

1. `status` 用于处理中状态
2. `message` 用于回复文本
3. `artifact` 用于图片和文件
4. `error` 用于异常
5. `done` 表示本轮结束

## 安全边界

第一阶段至少做这些：

1. 默认只监听 `127.0.0.1`
2. 如果需要手机访问，优先通过 `Tailscale`
3. 增加一个本地访问令牌
4. 不开放任意 shell 执行接口
5. 限制文件下载目录，只允许访问产物目录和明确白名单目录

## 与当前 TG 方案的关系

当前 TG 方案适合：

1. 快速发任务
2. 收结果
3. 收截图

未来 Web 方案适合：

1. 查看完整会话
2. 实时看状态
3. 管理多个会话
4. 更自然地看图片、代码块和文件

建议最终保留两套入口：

1. TG：轻量远程触发
2. Web：正式远程管理台

## 建议实施顺序

### 第 1 步

新建独立目录和基础后端服务。

### 第 2 步

把当前 TG 版的会话管理逻辑抽成公共模块。

### 第 3 步

做 Web 会话列表和详情页。

### 第 4 步

接入实时事件推送。

### 第 5 步

接入文件和图片展示。

### 第 6 步

再决定是否切换到 `codex app-server`。

## 我建议下一步直接做的最小范围

只做下面这些，不一口气铺太大：

1. 新目录 `codex-web-console`
2. 一个 Node 后端
3. 一个移动端会话页
4. 支持新建、续聊、查看状态、看图片

先把“手机上管理 Codex 会话”这件事跑通，再看是否追加审批、鉴权、历史搜索等能力。
